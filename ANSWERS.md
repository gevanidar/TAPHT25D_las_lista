# Frågor och svar

## Vad är skillnaden mellan enhetstest, integrationstest, regressionstest och prestandatest?

### Svar:

Ett enhetstest är gjort för att testa en specifik del at programmet. En del kan vara t.ex. en klass eller funktion. Enhetstester för funktioner testar att funktionen fungerar korrekt, och för klasser så kan man skapa enhetstester för hela klassen för att se till att klassens delar uppnår det som är avsett. Acceptans kriterier används för att särskilja vad som är korrekt implementation. Enhetstest för en klass som kan utföra addition, subtraktion, multiplikation och division kommer behöva verifiera att resultatet från varje funktion blir matematiskt korrekt. Det behöver också verifiera att det inte blir fel vid input, som t.ex. division med 0 ska inte orsaka att programmet kraschar - utan istället ge ett felmeddelande.
Integrationstester är tester för att se att de olika delarna av programmet fungerar tillsammans. Här testas sammankopplinger mellan olika funktioner eller klasser och där man ser till att de fungerar tillsammans. T.ex. så kan en klass för en kalkylator och ovan nämnda klass för matematiska beräkningar behövas testas tillsammans, en användare kan skriva in siffror och tecken i kalkylatorn. Matematik klassen hanterar berökningarna, men kopplingen mellan dem måste fungera korrekt. Verifiering att input bara innehåller siffror och tecken `+`, `-`, `*` samt `/` behöver göras. Det går inte bara att testa att kalkylatorn kan ta input och ge output till användaren, kopplingen måste testas.
Regressionstester är till för att se hur ändringar som görs i programmer inte påverkar den ursprungliga funktionaliteten. Om vi upptäcker en bugg i kalkylatorn ovan som tillåter att vi skickar in oönskade tecken och denna bug fixas, så kan regressiontestet se till att det gamla acceptans kriteriet för kalkylatorns funktion inte går sänder, t.ex. att man fortfarande kan skriva in de matematiska symbolerna..
Prestandatester är till för att se hur ändringar av delar av programmet påverkar prestandan. Prestanda tester kan användas för att se till att funktioner inte påverkas negativt av ändringar i funktionen, det är inte viktigt för prestanda tester att funktionen är korrekt (det sköter de andra testerna). De kan användas som en benchmark för att se till att ändringar i koden gör att koden förblir lika bra eller att den förbättras. Exempel på prestandatester är ofta baserade på hur lång tid det tar att utföra t.ex. en funktion, mängden minne som används vid en funktion kan också användas som ett prestandatest.



## Beskriv hur det går till när man arbetar med TDD.

### Svar:

Test drivern utveckling bygger på att man först skriver tester som är till för att verifiera att den kod man sedan skriver kommer att uppfylla krav eller acceptans kriterier.
En kund säger att de vill ha en ny funktion. T.ex. kan denna funktion vara att man ska kunna favoritmarkera böckerna i i listan på hemsidan och att man sedan ska kunna se vilka böcker man markerat som favoriter. Detta krav från kunden delas sedan in i User Stories, här ska det finnas en user story som hantera att man kan favoritmarkera en bok, en att man kan avmarkera favoriter och en där det finns en lista som användaren kan se över alla markerade favoritböcker. Det var inte explicit från kunden att man behöver kunna avmarkera favoriter, det kan vara bra att diskutera med kunden först om det är något som de vill ha eller inte (i detta fallet var det en väldigt enkel och nödvändig funktion). En user story skrivs på sätter `Som en ..., vill jag ..., så att jag ...`. Exempel `Som en användare, vill jag kunna favoritmarkera en bok, så att jag kan hålla ordning på mina favoritböcker`. Det skrivs också acceptanskriterier för funktionen för att man konkret ska kunna verifiera att implementationen av den nya funktionen blir korrekt. En user story tas upp av en testare som skriver ett eller flera tester för att säkerställa att funktionen blir rätt. Det viktiga steget här är att testerna inte går igenom (det blir rött) vid första körningen av testerna, eftersom funktionen inte är utvecklad så ska den inte fungera, om ett test hade gått igenom direkt så går det inte att säkerställa att utvecklingen av den nya funktionen gjort att testet gått igenom. En utvecklare skriver sedan kod som gör att testet går igenom, dvs går från rött till grönt. När utvecklaren har gjort sin första implementation så går hen vidare med att se till att refaktorisera och se till att funktionen upphåller de standarder eller regler man har på företaget. När funktionen fungerar men är långsam prestandatester vara bra att använda (då blir det en ny tdd cykel). När utvecklaren är klar med att fixa allt så skickas koden vidare till granskning (det är utanför TDD cykeln men industristandard).

## Beskriv hur BDD skiljer sig från TDD.

### Svar:

Behaivour Driven Development är en utvecklingstyp där man använder sig av skriftspråk för att beskriva tester. Det betyder att istället för att skriva t.ex. `assert a == b` så beskriver man hur man vill att programmet ska fungera (eller en del av programmet). Detta gör att en person som inte är utvecklare kan förstå vad testet gör, det är över enklare att resonera över vad testet gör. T.ex. så kan man använda BDD när man utvecklar webbsidor och man behöver navigera runt mellan sidorna och klicka på olika saker och verifiera att det ser rätt ut och att sidan innehåller det man tänker att den ska innehålla. Om man jämför det mot TDD så kan man säga att TDD ofta testar att koden fungerar och BDD testar att den som man vill kunna göra kan man göra. Om man skulle simulera att gå och handla så hade TDD sett till att det går att komma in i affären, välja olika varor och sedan att betalning fungerar. Vid BDD så hade man skapat olika scenario som skulle kunna inträffa, BDD beskrivs med acceptans kriterier `Given`, `When`, `Then`. `Given` beskriver förutsättningarna t.ex. att jag är på webbsidan och är inloggad som en användare, `When` beskriver det jag gör på webbsidan t.ex. att jag går till sökfältet och söker efter "Strumpor". `Then` är det förväntade resultatet utifrån de två tidigare stegen, t.ex. att det syns en lista över "Strumpor" och att listan innehåller 5 resultat. 

## Tänk dig att du skulle göra en webbsida som liknar Läslistan, både frontend och backend. Om du fick välja förutsättningslöst, vilka sorters tester skulle du vilja använda? Motivera ditt val.

### Svar:

Enhetstester behövs för att se till att alla delar av programmet fungerar, det gör det enklare att lokalisera vad det blir fel om någon ändring införs som orsakar att någon funktion slutar fungera. Integrationstester är även bra för att kunna se till att samspelet mellan de olika delarna av programmer fungerar, man kan tänka sig att enhetstesterna går igenom men att kommunikationen mellan delarna går sönder, om alla enhetstester är gröna men koden är sönder så kan det vara svårt att veta var det är fel - integrationstesterna kan visa mellan vilka delar av programmet som det blir fel. Vid uppskalning så hade prestandatester kunnat vara bra för att se till att det inte tar för lång tid att generera exempelvis en lista som ska skickas till frontend eftersom det hade gett en dålig användarupplevelse, i fallet av läslista så hade det kanske varit ohållbart att ta in alla bäcker i hela världen och visa favoritlistor för alla människor då hade smartare implementation behövts och det hade prestandatester kunnat användas till. Regressionstester hade varit bra för att se till att funktioner i backend inte ändras och påverkar frontend och samma för frontend, om backend skickar en lista av böcker till frontend med frontend numera läser in en `map` så kommer programmet att krascha. Det hade regressionstester kunnat vara användbara till. Tester med BDD blir också användbart då funktioner på flera sidor ska fungera, man ska kunna navigera fram och tillbaka och det ska även fungera i en webbläsare, olika användar scenario kommer behövas testas för att man ska se till att det går att använda sidan så som det är tänkt. Det blir också en beskrivning av hur systemet är tänkt att användas, som är lättläslig för både utvecklare och icke programmerings kunniga. Beskrivningen av hur systemet är tänkt att användas är också väldigt användbart för att kunna få en översikt i vad det finns för funktionalitet och för att kunna sätta sig in i avsikten med programmet.
